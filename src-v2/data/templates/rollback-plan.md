## Rollback Steps

### Step 1: Announce Rollback

```bash
# Post to incident channel
./scripts/announce-rollback.sh --release {{release_id}} --reason "{{reason}}"
```

### Step 2: Stop Deployment Pipeline

```bash
# Halt any in-progress deployments
./scripts/halt-deployments.sh
```

### Step 3: Execute Rollback

<!-- FILL: Document rollback commands -->
**Application Rollback:**
```bash
# Rollback to previous version
kubectl rollout undo deployment/{{service}} -n production

# Or deploy specific version
kubectl set image deployment/{{service}} \
  {{container}}={{image}}:{{rollback_version}} -n production
```

**Database Rollback (if needed):**
```bash
# Execute rollback migration
./scripts/db-rollback.sh --to-version {{db_version}}
```

### Step 4: Verify Rollback

```bash
# Health check
./scripts/health-check.sh --service {{service}}

# Verify version
kubectl get deployment {{service}} -o jsonpath='{.spec.template.spec.containers[0].image}'
```

## Rollback Scenarios

### Scenario A: Application Only

**Duration:** ~5 minutes

| Step | Action | Command |
|------|--------|---------|
| 1 | Rollback deployment | `kubectl rollout undo` |
| 2 | Verify pods healthy | `kubectl get pods` |
| 3 | Verify endpoints | Health check |

### Scenario B: Application + Database

**Duration:** ~15 minutes

| Step | Action | Command |
|------|--------|---------|
| 1 | Stop traffic | Enable maintenance mode |
| 2 | Rollback database | Migration script |
| 3 | Rollback application | `kubectl rollout undo` |
| 4 | Verify data integrity | Integrity check |
| 5 | Resume traffic | Disable maintenance mode |

### Scenario C: Multi-Service Rollback

**Duration:** ~30 minutes

| Step | Action | Services |
|------|--------|----------|
| 1 | Stop dependent services | Service B, C |
| 2 | Rollback core service | Service A |
| 3 | Rollback dependent services | Service B, C |
| 4 | Verify integration | E2E test |
