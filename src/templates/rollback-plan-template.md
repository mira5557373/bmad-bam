## Rollback Strategies

### Strategy A: Feature Flag Disable

**Use when:** Changes are behind feature flags, no database changes

| Step | Action | Command | Verify |
| ---- | ------ | ------- | ------ |
| 1 | Disable feature flag | `flag set {{feature_flag}} false` | Flag status |
| 2 | Monitor error rates | — | Error dashboard |
| 3 | Verify functionality | — | Smoke tests |

**Rollback Time:** ~1 minute
**Data Impact:** None

### Strategy B: Blue-Green Switch

**Use when:** Full deployment switch needed, stateless services

| Step | Action | Command | Verify |
| ---- | ------ | ------- | ------ |
| 1 | Switch traffic to blue | `kubectl set route blue` | Traffic routing |
| 2 | Drain green connections | `kubectl drain green` | Connection count |
| 3 | Monitor metrics | — | Dashboard |
| 4 | Scale down green | `kubectl scale green --replicas=0` | Pod count |

**Rollback Time:** ~5 minutes
**Data Impact:** None

### Strategy C: Rolling Revert

**Use when:** Gradual rollback needed, canary deployment

| Step | Action | Command | Verify |
| ---- | ------ | ------- | ------ |
| 1 | Update deployment image | `kubectl set image ... :{{previous_version}}` | Image version |
| 2 | Monitor rollout | `kubectl rollout status` | Rollout progress |
| 3 | Verify pod health | `kubectl get pods` | Pod status |
| 4 | Run smoke tests | — | Test results |

**Rollback Time:** ~10-15 minutes
**Data Impact:** None

### Strategy D: Database Rollback

**Use when:** Database schema changes need reversal

| Step | Action | Command | Verify |
| ---- | ------ | ------- | ------ |
| 1 | Set maintenance mode | `./maintenance-mode enable` | Status page |
| 2 | Stop application | `kubectl scale --replicas=0` | Pod count |
| 3 | Run down migration | `./migrate down {{migration_version}}` | Migration status |
| 4 | Verify schema | `./verify-schema` | Schema check |
| 5 | Deploy previous version | `kubectl set image ... :{{previous_version}}` | Image version |
| 6 | Start application | `kubectl scale --replicas={{replicas}}` | Pod count |
| 7 | Disable maintenance mode | `./maintenance-mode disable` | Status page |

**Rollback Time:** ~30-60 minutes
**Data Impact:** Potential data loss for new schema fields
