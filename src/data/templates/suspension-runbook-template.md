## State Machine

### Tenant States

| State | Description | Allowed Transitions |
| ----- | ----------- | ------------------- |
| ACTIVE | Full platform access | SUSPENDED |
| SUSPENDED | Access revoked, data preserved | REACTIVATED, ARCHIVED |
| REACTIVATED | Restored from suspension | ACTIVE |
| ARCHIVED | Permanently deactivated | None (terminal) |

## Suspension Procedure

### Pre-Suspension Checklist

- [ ] Verify suspension trigger is valid
- [ ] Confirm grace period has elapsed (if applicable)
- [ ] Document suspension reason
- [ ] Prepare notification content

### Step 1: Initiate Suspension

**Duration:** < 1 minute

**Actions:**
1. Update tenant state to SUSPENDED
2. Record suspension timestamp and reason
3. Log audit event

### Step 2: Revoke Access

**Duration:** < 30 seconds

**Order of Operations:**
1. Invalidate all user sessions
2. Block API keys
3. Terminate running agents
4. Update RLS policies to deny access
5. Clear tenant cache

### Step 3: Stop Background Services

**Duration:** < 1 minute

**Actions:**
1. Pause scheduled jobs
2. Suspend webhook delivery
3. Skip tenant messages in queue
4. Terminate agent executions

### Step 4: Send Notifications

**Actions:**
1. Send suspension email to tenant admin
2. Send in-app notification (if accessible)
3. Log notification delivery

## Grace Period Handling

### Grace Period Matrix

| Trigger | Grace Period | Warnings |
| ------- | ------------ | -------- |
| Billing Failure | {{billing_grace_days}} days | Days -7, -3, -1 |
| Policy Violation (Minor) | 3 days | Immediate, day 2 |
| Inactivity | 30 days | Days -30, -20, -10, -3 |

### Feature Degradation During Grace Period

| Level | When | Restrictions |
| ----- | ---- | ------------ |
| Warning | Grace period start | No new agent creation |
| Restricted | Day 5 | Read-only access |
| Suspended | Grace period end | Full access revoked |

## Reactivation Procedure

### Reactivation Triggers

- Payment received and verified
- Appeal approved
- Admin manual reactivation

### Step 1: Verify Reactivation Eligibility

- [ ] Payment confirmed (if billing issue)
- [ ] Appeal approved (if policy violation)
- [ ] Admin authorization (if manual)

### Step 2: Restore Access

**Order of Operations:**
1. Update tenant state to REACTIVATED
2. Re-enable RLS policies
3. Restore API key access
4. Enable user sessions
5. Resume scheduled jobs

### Step 3: Resume Services

**Actions:**
1. Resume webhook delivery
2. Re-enable queue processing
3. Allow agent execution
4. Clear any degradation flags

### Step 4: Send Notifications

**Actions:**
1. Send reactivation email to all users
2. Send in-app welcome back notification
3. Log reactivation event

### Step 5: Transition to Active

**Actions:**
1. Update state from REACTIVATED to ACTIVE
2. Clear suspension history flags
3. Resume normal monitoring

## Troubleshooting

### Issue: Suspension Failed to Complete

**Symptoms:** Tenant still has partial access

**Resolution:**
1. Check suspension status in database
2. Verify all revocation steps completed
3. Manually revoke any remaining access
4. Re-run suspension procedure

### Issue: Reactivation Failed

**Symptoms:** Tenant cannot access after reactivation

**Resolution:**
1. Verify state is ACTIVE
2. Check RLS policies restored
3. Clear cache and retry
4. Check for orphaned session invalidations
