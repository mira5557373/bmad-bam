| Field | Type | Required | Description |

| hierarchy_path | LTREE/Array | Yes | Materialized path for queries |
| hierarchy_level | Integer | Yes | Depth in hierarchy (0 = root) |
| created_at | Timestamp | Yes | Creation timestamp |
| created_by | UUID | Yes | Creating user/system |

### 2.2 Materialized Path Pattern

```
Path format: root_id.child_id.grandchild_id

Examples:
  - Root tenant:      "tenant_abc"
  - Child:            "tenant_abc.tenant_def"
  - Grandchild:       "tenant_abc.tenant_def.tenant_ghi"

Query patterns:
  - Get all descendants: path LIKE 'tenant_abc.%'
  - Get all ancestors:   'tenant_abc.tenant_def.tenant_ghi' LIKE path || '%'
  - Get siblings:        parent_tenant_id = 'tenant_def'
```

### 2.3 Adjacency vs Materialized Path Trade-offs

| Approach | Insert/Update | Read Ancestors | Read Descendants | Move Subtree |
|----------|---------------|----------------|------------------|--------------|
| Adjacency List | O(1) | O(n) | O(n) recursive | O(1) |
| Materialized Path | O(1) | O(1) | O(log n) | O(n) updates |
| Closure Table | O(n) | O(1) | O(1) | O(n^2) |
| Nested Sets | O(n) | O(1) | O(1) | O(n) |

### 3.3 Permission Resolution Algorithm

```
┌─────────────────────────────────────────────────────────────────┐
│              Permission Resolution Flow                          │
│                                                                  │
│  1. Start at target tenant                                       │
│  2. Load tenant's explicit permissions                           │
│  3. Walk up hierarchy to root                                    │
│     ┌──────────────────────────────────────┐                    │
│     │ For each ancestor:                    │                    │
│     │   - Apply inheritance rules           │                    │
│     │   - Merge/override based on model     │                    │
│     │   - Record permission source          │                    │
│     └──────────────────────────────────────┘                    │
│  4. Apply platform-level defaults                                │
│  5. Return effective permission set                              │
│                                                                  │
│  Effective = f(explicit, inherited, defaults)                    │
└─────────────────────────────────────────────────────────────────┘
```

## Tenant Lifecycle in Hierarchy

### 6.1 Tenant Creation

| Step | Action | Validation |
|------|--------|------------|
| 1 | Receive create request with parent_id | Parent exists and active |
| 2 | Verify parent can create children | Permission check |
| 3 | Check hierarchy depth limit | Max depth not exceeded |
| 4 | Check sibling count limit | Max children not exceeded |
| 5 | Generate tenant_id and hierarchy_path | Unique, valid path |
| 6 | Inherit permissions from parent | Apply inheritance model |
| 7 | Allocate quota from parent pool | Within available quota |
| 8 | Create tenant record | Transactional |
| 9 | Notify parent admin | Async notification |

### 6.2 Tenant Movement (Re-parenting)

| Step | Action | Validation |
|------|--------|------------|
| 1 | Request move from old_parent to new_parent | Both parents exist |
| 2 | Verify authorization | Admin of both branches |
| 3 | Check new hierarchy constraints | Depth, siblings |
| 4 | Calculate new hierarchy_path | No circular reference |
| 5 | Update all descendant paths | Batch update |
| 6 | Recalculate permissions | Apply new inheritance |
| 7 | Rebalance quotas | Source and destination |
| 8 | Update billing relationships | Cost center changes |
| 9 | Audit log movement | Complete trail |

## Query Patterns

### 7.1 Common Hierarchy Queries

| Query | Description | Index Recommendation |
|-------|-------------|---------------------|
| Get all ancestors | Find path to root | hierarchy_path GIST |
| Get all descendants | Find all children recursively | hierarchy_path prefix |
| Get direct children | Immediate children only | parent_tenant_id B-tree |
| Get siblings | Same parent | parent_tenant_id B-tree |
| Get subtree depth | Levels below tenant | hierarchy_level |
| Find by path | Exact path match | hierarchy_path exact |

### 7.2 Performance Considerations

| Operation | Complexity | Optimization |
|-----------|------------|--------------|
| Ancestor lookup | O(depth) | Materialized path + cache |
| Descendant count | O(n) | Denormalized counter |
| Permission resolution | O(depth) | Permission cache |
| Billing rollup | O(n) | Background aggregation |
| Subtree move | O(descendants) | Batch updates, off-peak |

### 7.3 Caching Strategy

| Level | Isolation Method | Cross-Access |
|-------|------------------|--------------|
| Root | Separate schema/database | Platform admin only |
| Branch | RLS with hierarchy filter | Parent + own descendants |
| Leaf | RLS tenant_id | Own data only |

### 8.2 RLS Policy for Hierarchy

```
Policy considerations:
  - tenant_id IN (SELECT descendant_ids(current_tenant_id))
  - OR tenant_id = current_tenant_id
  - OR (has_cross_tenant_permission AND in_same_hierarchy)

Index support:
  - hierarchy_path for ancestor/descendant queries
  - parent_tenant_id for direct relationships
  - tenant_id for exact matches
```
