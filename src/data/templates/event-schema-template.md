## Event Naming Convention

### Pattern

```
{domain}.{entity}.{action}.{version}
```

### Examples

| Event Name                          | Domain   | Entity   | Action   | Version |
| ----------------------------------- | -------- | -------- | -------- | ------- |
| billing.invoice.created.v1          | billing  | invoice  | created  | v1      |
| user.account.activated.v2           | user     | account  | activated| v2      |
| agent.run.completed.v1              | agent    | run      | completed| v1      |

### Action Types

| Action      | Description                                    |
| ----------- | ---------------------------------------------- |
| created     | New entity was created                         |
| updated     | Entity was modified                            |
| deleted     | Entity was removed (soft or hard)              |
| activated   | Entity transitioned to active state            |
| deactivated | Entity transitioned to inactive state          |
| completed   | Process or workflow completed                  |
| failed      | Process or workflow failed                     |
| requested   | Action was requested (async)                   |
| approved    | Action was approved                            |
| rejected    | Action was rejected                            |

## Event Envelope Schema

### Standard Event Envelope

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["event_id", "event_type", "event_version", "tenant_id", "timestamp", "payload"],
  "properties": {
    "event_id": {
      "type": "string",
      "format": "uuid",
      "description": "Unique event identifier"
    },
    "event_type": {
      "type": "string",
      "pattern": "^[a-z]+\\.[a-z]+\\.[a-z]+\\.v[0-9]+$",
      "description": "Event type following naming convention"
    },
    "event_version": {
      "type": "string",
      "pattern": "^v[0-9]+$",
      "description": "Schema version"
    },
    "tenant_id": {
      "type": "string",
      "format": "uuid",
      "description": "Tenant identifier for multi-tenant isolation"
    },
    "correlation_id": {
      "type": "string",
      "format": "uuid",
      "description": "Correlation ID for request tracing"
    },
    "causation_id": {
      "type": "string",
      "format": "uuid",
      "description": "ID of event that caused this event"
    },
    "timestamp": {
      "type": "string",
      "format": "date-time",
      "description": "Event creation timestamp (ISO 8601)"
    },
    "source": {
      "type": "string",
      "description": "Service/module that produced the event"
    },
    "metadata": {
      "type": "object",
      "properties": {
        "user_id": {
          "type": "string",
          "format": "uuid"
        },
        "request_id": {
          "type": "string"
        },
        "tier": {
          "type": "string",
          "enum": ["FREE", "PRO", "ENTERPRISE"]
        }
      }
    },
    "payload": {
      "type": "object",
      "description": "Event-specific payload data"
    }
  }
}
```

```typescript
// Check before processing
const isDuplicate = await idempotencyStore.exists(event.event_id, event.tenant_id);
if (isDuplicate) {
  return previousResult;
}

  result_hash: hash(result)
});
```

## Event Versioning

### Schema Evolution Strategy

| Evolution Type       | Backward Compatible | Forward Compatible | Handling               |
| -------------------- | ------------------- | ------------------ | ---------------------- |
| Add optional field   | Yes                 | Yes                | No version bump needed |
| Add required field   | No                  | Yes                | Major version bump     |
| Remove field         | Yes                 | No                 | Minor version bump     |
| Rename field         | No                  | No                 | Major version bump     |
| Change field type    | No                  | No                 | Major version bump     |

### Version Coexistence

```yaml
# Consumers MUST handle multiple versions during migration
event_handlers:
  - event_type: "{{domain}}.{{entity}}.{{action}}.v1"
    handler: handleV1Event
    deprecated: true
    sunset_date: "{{v1_sunset}}"
  
  - event_type: "{{domain}}.{{entity}}.{{action}}.v2"
    handler: handleV2Event
    current: true
```

### Migration Path

| From Version | To Version | Migration Steps                      |
| ------------ | ---------- | ------------------------------------ |
| v1           | v2         | 1. Deploy v2 handlers<br>2. Enable dual-writing<br>3. Migrate consumers<br>4. Disable v1 |

## Multi-Tenant Events

### Tenant Context in Events

| Field          | Required | Description                          |
| -------------- | -------- | ------------------------------------ |
| tenant_id      | Yes      | Tenant identifier in envelope        |
| metadata.tier  | Yes      | Tenant tier for routing/filtering    |

### Tenant Isolation Guarantees

| Aspect                  | Enforcement                          |
| ----------------------- | ------------------------------------ |
| Event Visibility        | Events filtered by tenant_id         |
| Consumer Routing        | Tenant-specific consumer groups      |
| Dead Letter Queues      | Separate DLQ per tenant tier         |
| Event Retention         | Tier-based retention policies        |

### Tenant-Aware Event Routing

```yaml
routing_rules:
  - tenant_tier: FREE
    topic_suffix: "-free"
    max_throughput: 100/sec
    retention_days: 7
  
  - tenant_tier: PRO
    topic_suffix: "-pro"
    max_throughput: 1000/sec
    retention_days: 30
  
  - tenant_tier: ENTERPRISE
    topic_suffix: "-enterprise"
    max_throughput: 10000/sec
    retention_days: 90
    dedicated_partition: true
```

### Cross-Tenant Event Rules

| Scenario                    | Allowed | Handling                          |
| --------------------------- | ------- | --------------------------------- |
| Event within same tenant    | Yes     | Normal processing                 |
| Event across tenants        | No      | Rejected at consumer validation   |
| System/platform events      | Yes     | No tenant_id, special routing     |
| Admin cross-tenant events   | Yes     | Requires elevated permissions     |

## Event Processing Guarantees

### Delivery Semantics

| Guarantee          | Implementation                       |
| ------------------ | ------------------------------------ |
| At-least-once      | Consumer acknowledgment after commit |
| Ordering           | Partition key = tenant_id + entity_id|
| Durability         | Replication factor >= 3              |

### Retry Policy

| Attempt | Delay        | Action                              |
| ------- | ------------ | ----------------------------------- |
| 1       | Immediate    | Process event                       |
| 2       | 1 second     | Retry with exponential backoff      |
| 3       | 5 seconds    | Retry with exponential backoff      |
| 4       | 30 seconds   | Retry with exponential backoff      |
| 5       | 5 minutes    | Move to dead letter queue           |

### Dead Letter Queue Handling

```yaml
dlq_config:
  topic: "{{module_name}}.dlq"
```
