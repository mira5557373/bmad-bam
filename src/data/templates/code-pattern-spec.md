## Repository Pattern

### Base Repository Interface

```
TenantScopedRepository<T>:
  - find_by_id(ctx: TenantContext, id: UUID) -> T?
  - find_all(ctx: TenantContext, filters: Filters, page: Pagination) -> Page<T>
  - save(ctx: TenantContext, entity: T) -> T
  - delete(ctx: TenantContext, id: UUID) -> bool
  - exists(ctx: TenantContext, id: UUID) -> bool
```

### Implementation Requirements

| Requirement | Implementation |
|-------------|----------------|
| Tenant filtering | Set RLS before every query |
| Validation | Verify tenant_id matches context |
| Audit | Log all write operations |
| Connection | Use connection pool with tenant scope |

## Service Layer Pattern

### Domain Service

| Characteristic | Requirement |
|----------------|-------------|
| Purpose | Business logic spanning entities |
| Dependencies | Domain entities only |
| Tenant awareness | None (pure domain logic) |
| Side effects | None |

### Application Service

| Characteristic | Requirement |
|----------------|-------------|
| Purpose | Use case orchestration |
| Dependencies | Domain services, repositories, facades |
| Tenant awareness | TenantContext required |
| Side effects | Database, events, external calls |

## Error Handling Pattern

### Error Hierarchy

```
FacadeError (base)
├── ValidationError
│   ├── InvalidInputError
│   └── MissingFieldError
├── NotFoundError
│   └── EntityNotFoundError
├── ConflictError
│   └── DuplicateEntityError
├── AuthorizationError
│   └── TenantMismatchError
└── InternalError
    └── ServiceUnavailableError
```

### Error Response Format

```json
{
  "error": {
    "code": "{{error_code}}",
    "message": "Human-readable message",
    "details": {},
    "correlation_id": "uuid"
  }
}
```

## Logging Pattern

### Structured Log Format

| Field | Required | Source |
|-------|----------|--------|
| timestamp | Yes | System |
| level | Yes | Logger |
| message | Yes | Code |
| tenant_id | Yes* | TenantContext |
| user_id | No | TenantContext |
| correlation_id | Yes | Request |
| service | Yes | Config |
| module | No | Code location |

### Log Levels

| Level | Use Case |
|-------|----------|
| DEBUG | Development tracing |
| INFO | Business events |
| WARN | Recoverable issues |
| ERROR | Failures |

## Event Pattern

### Event Structure

```
DomainEvent:
  - event_id: UUID
  - event_type: "{module}.{entity}.{past_tense_verb}"
  - tenant_id: UUID
  - aggregate_id: UUID
  - occurred_at: DateTime
  - payload: JSON
```

### Event Publishing

| Rule | Implementation |
|------|----------------|
| Tenant mandatory | Always include tenant_id |
| Idempotency | Include event_id for dedup |
| Ordering | Per-aggregate ordering only |
| Reliability | At-least-once delivery |

## API Pattern

### Endpoint Structure

```
/api/v1/{resource}           # Collection
/api/v1/{resource}/{id}      # Single resource
/api/v1/{resource}/{id}/{sub} # Nested resource
```

### Response Format

```json
{
  "data": {},
  "meta": {
    "pagination": {},
    "timestamp": "iso8601"
  }
}
```

## Testing Pattern

### Test Categories

| Category | Purpose | Location |
|----------|---------|----------|
| Unit | Domain logic | `tests/unit/` |
| Integration | Database, external | `tests/integration/` |
| Contract | Facade compliance | `tests/contract/` |
| E2E | Full flow | `tests/e2e/` |

### Multi-Tenant Test Requirements

- [ ] All tests use tenant fixtures
- [ ] Isolation tests with 2+ tenants
- [ ] Tier-specific behavior tested
- [ ] No hardcoded tenant IDs
